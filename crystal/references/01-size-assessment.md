# 01 · 規模評估（Phase 0）

> **本檔對應**：SKILL.md Phase 0 與「何時使用」規模前置表。
> **沿原作**：S/M/L 判準、計數指令、工具決策分級皆沿 v2.5 原設計；「LLM 擅長理解小型程式碼、工具是增強非取代」核心句保留（見 §3）。

---

## 1. 為什麼要先評估規模

知識管理工具本身有啟動成本（安裝、索引、維護）。小專案上重工具是淨負擔。Phase 0 用「重要程式碼檔案數量」把成本意識寫進流程：先數檔，再決定後面要不要動 graphify 這類工具。這是整條管線最務實的一步，必須先做。

## 2. 計數指令

只數「重要程式碼檔案」，副檔名限定下列集合（依專案技術棧增減）：

```bash
# 在 repos/<repo>/ 底下數，排除依賴目錄
FILE_COUNT=$(find repos -type f \
  \( -name '*.py' -o -name '*.ts' -o -name '*.js' \
     -o -name '*.cpp' -o -name '*.h' -o -name '*.java' \
     -o -name '*.go' -o -name '*.rs' \) \
  -not -path '*/node_modules/*' \
  -not -path '*/.venv/*' \
  -not -path '*/vendor/*' \
  -not -path '*/dist/*' \
  | wc -l)
echo "重要程式碼檔案數：$FILE_COUNT"
```

計數只含產品碼；測試檔、產生物、依賴一律排除（上例已排 node_modules/.venv/vendor/dist，依實際專案補排除路徑）。

## 3. S/M/L 判準與工具決策

**核心原則（原作原文精神保留）**：LLM 本身就擅長理解小型程式碼的關聯性。工具是「增強」而非「取代」。

| 規模 | 檔數 | Phase 3 依賴圖決策 | 理由 |
|---|---|---|---|
| **S（Small）** | <5 檔 | **完全跳過工具**。不建知識庫 | LLM 直讀即懂，全套流程是淨負擔 |
| **M（Medium）** | 5–20 檔 | **局部 graphify**：只對 hub 模組/跨界介面/外部依賴密集處跑 | 大部分靠 LLM 讀，工具補在關鍵節點 |
| **L（Large）** | >20 檔 | **全套**：graphify + 依賴圖索引工具跑全 repo | 超出 LLM 單次可靠掌握範圍，需結構化索引 |

判定落在邊界（如剛好 5 檔或 20 檔）時，看模組耦合度：耦合高、跨檔呼叫密就往上一級靠。

## 4. 各層/各 Phase 的工具決策速查

| 層/動作 | S | M | L |
|---|---|---|---|
| raw/ 證據保全 | 需要 | 需要 | 需要 |
| repos/ 盤點 | 需要 | 需要 | 需要 |
| graphify 依賴圖 | 跳過 | 局部（hub） | 全套 |
| 依賴圖索引工具 | 不用 | 視需要 | 標配 |
| wiki 策展 | 輕量（少數頁） | 需要 | 需要 |
| spec 契約 | 視需要 | 需要 | 需要 |

工具的具體用途分工、安裝與版本釘選見 `03-graphify-tooling.md`。

## 5. 規模評估的輸出

Phase 0 結束時應在 raw/ 留一筆評估記錄（一份 `YYYY-MM-DD-size-assessment.md`），至少含：檔數實測值、判定規模（S/M/L）、據此決定的 Phase 3 工具策略。這筆記錄本身是證據，走 raw/ 的 append-only 規則（見 `02-evidence-and-code.md`）。
